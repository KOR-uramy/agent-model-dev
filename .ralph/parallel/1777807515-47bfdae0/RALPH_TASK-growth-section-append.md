## 축별 갭 (우리가 부족할 수 있는 점 — bullet)

- **UI**
  - GitHub 대시보드 수준의 **전역 내비·컨텍스트 스위처(워크스페이스/조직)** 가 **한 눈에 보이는 위치**에 고정되어 있지 않을 수 있다.
  - Intercom식 **히어로 → 사회적 증거 → CTA** 리듬이 랜딩·인앱 첫 화면에 **동일한 세로 리듬**으로 맞지 않을 수 있다.
- **UX**
  - API 키 **회전·폐기·스코프** 설명이 GitHub PAT 가이드 대비 **단계별 체크리스트**로 부족할 수 있다.
  - 수집 실패·지연 시 **재시도·멱등 키** 안내가 Datadog/GitHub webhook 문서 대비 **한 화면 요약**으로 부족할 수 있다.
- **디자인**
  - **성공/경고/오류** 색의 의미가 화면마다 달라 Datadog·Intercom의 **시맨틱 컬러** 대비 일관성이 약할 수 있다.
  - 마케팅과 앱의 **타이포 스케일**이 Intercom 랜딩 대비 **단계 수가 적어** 위계가 평면적일 수 있다.
- **수익화**
  - **무료 한도·리셋 주기·초과 시 동작**이 GitHub Actions minutes처럼 **숫자+주기**로 사용자에게 고정 노출되지 않을 수 있다.
  - 엔터프라이즈/팀 플랜 **문의 vs 셀프서브** 경계가 카피·CTA에 분명하지 않을 수 있다(Intercom 대비).
- **트래픽·운영**
  - Webhook/수집 엔드포인트의 **서명 검증·재전송·타임아웃**이 운영자용 **짧은 런북(표 1개)** 으로 묶여 있지 않을 수 있다.
  - **상태/인시던트** 공개 페이지나 배너가 Datadog status 대비 사용자 신뢰 축에서 비어 있을 수 있다.
- **마케팅**
  - **보안·컴플라이언스(SOC2 등)** 링크가 푸터·랜딩에 **개수·순서**가 정해져 있지 않을 수 있다(Intercom·Datadog 신뢰 블록 대비).
  - “에이전트 관측”이 아닌 **구매자 한 줄 가치**가 히어로에 고정되어 있지 않을 수 있다.
