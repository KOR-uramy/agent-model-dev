## 축별 갭 (우리가 부족할 수 있는 점 — bullet)

### UI

- Sentry 대비 **프로젝트(워크스페이스) 스코프 전환·브레드크럼**이 상단에 고정돼 있지 않으면 멀티 워크스페이스 사용자가 길을 잃기 쉽다.
- Honeycomb/Axiom식 **데이터 탐색(쿼리·필터) 진입점**이 대시보드에 없으면 “수집만 되고 분석 UI가 얇다”는 인상이 남을 수 있다.

### UX

- 수집 실패·**버스트 후 대기** 시나리오가 Sentry SDK 문서 수준의 **한 화면 플레이북**(코드 스니펫 + 기대 헤더)으로 묶여 있지 않으면 `docs`와 인앱이 갈린다.
- **첫 쿼리/첫 이벤트 가시화**까지의 단계 수가 Honeycomb Quickstart 대비 많거나 README와 화면 번호가 어긋나면 이탈이 커진다.

### 디자인

- 관측 제품은 **다크 모드·코드 블록·밀도**가 브랜드의 일부인데, 랜딩과 앱의 **톤·간격 스케일**이 한 세트로 고정돼 있지 않으면 “도구감”이 약해진다.
- **상태 색(경고/치명/성공)** 의 의미가 화면마다 다르면 Sentry Issue 심각도 UI 대비 신뢰가 떨어진다.

### 수익화

- **무료 볼륨·보존 기간·초과 시 동작**이 가격 페이지·인앱·API 한도 JSON에 **숫자 단위까지** 맞지 않으면 Axiom/PostHog류 PLG 대비 전환 전 질문이 남는다.
- **청구 실패·카드 갱신** 안내가 결제 플로 밖에만 있고 대시보드 배너가 없으면 B2B SaaS 관례 대비 갭이다.

### 트래픽·운영

- **버스트 허용·슬라이딩 윈도** 설명이 공개 문서에 없으면 Honeycomb 대비 “왜 갑자기 429인지” 설득이 약하다(내부 구현만으로는 부족).
- **멱등·중복 제거** 키가 없으면 클라이언트 재시도 시 Sentry event ID 패턴 대비 데이터 품질이 떨어진다.

### 마케팅

- **보안·신뢰·상태** 링크가 푸터에 없으면 엔터프라이즈 랜딩(Sentry Trust) 대비 첫인상이 약하다.
- **“무엇을 남기는가”(데이터 보존·삭제)** 한 문장이 히어로 근처에 없으면 규제 민감 팀이 이탈한다.
