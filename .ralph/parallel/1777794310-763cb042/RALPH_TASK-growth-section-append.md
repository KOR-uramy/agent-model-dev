## 우리가 부족한 점 (축별 bullet)

### UI

- 핵심 플로우 화면의 **정보 계층**(무엇이 1차인지)이 Langfuse/PostHog 대비 덜 압축되어 있거나, 반대로 **빈 공간**이 많아 “다음 행동”이 흐릿할 수 있다.
- **테이블·카드·필터** 패턴이 동종 대비 일관된 그리드/밀도 규칙으로 묶여 있지 않을 수 있다.

### UX

- **에러·빈 상태·로딩**에서 “지금 무엇을 하면 되는지” 한 줄 가이드가 동종 대비 적을 수 있다.
- 폼 검증 실패 시 **필드 단위 복구 경로**(포커스·스크롤)가 약할 수 있다.

### 디자인

- **타이포 스케일·간격 토큰**이 컴포넌트마다 미세하게 달라 브랜드 일관성이 약해 보일 수 있다.
- 다크/라이트·강조색이 **신뢰·전문성** 측면에서 경쟁 대비 덜 정돈될 수 있다.

### 수익화

- **플랜 한도·가격·결제 실패** 메시지가 UI 카피·문서·환경 변수 주석과 **한 소스**로 맞춰져 있는지 검증 단계가 부족할 수 있다.
- 무료 체험/업그레이드 **전환 문구**가 PostHog 수준의 명확함에 못 미칠 수 있다.

### 트래픽·운영

- 수집 API **레이트·쿼터·413/429** 사용자 대면 메시지가 Axiom 대비 문서·인앱에서 덜 노출될 수 있다.
- **관측·알림**(실패율, 급증)이 운영자에게 닿는 경로가 제한적일 수 있다.

### 마케팅

- 랜딩·README의 **가치 제안 한 문장**이 경쟁 대비 덜 압축되었거나, **사회적 증거**(고객 로고, 사용 사례)가 부족할 수 있다.
- 문서·인앱에서 **다음 단계 CTA**(키 발급, 첫 이벤트) 연결이 분산될 수 있다.

## 우선순위 높은 순 — 측정 가능한 `- [ ]` 항목 (본 절에 추가)

아래는 각 항목이 **완료 정의가 숫자·산출물·URL로 검증 가능**하도록 썼다. 오케스트레이터는 기존 6개 축 `- [ ]`는 유지하고, **하위 항목** 또는 **독립 `- [ ]`**로 추가하면 된다.

- [ ] **UI** — `/`, `/login`, `/register`, `/dashboard`, 워크스페이스 상세 **5화면**에 대해 Langfuse·PostHog **대응 URL**(공개 데모 또는 문서 스크린샷 섹션)을 1줄씩 적은 **비교 표**를 `.ralph/parallel/` 하위 등 **README·package 충돌 구역 밖** 마크다운 파일에 두고, 화면별 **레이아웃·밀도·1차 CTA 위치** 차이를 화면당 **최소 2문장**으로 기록한다.
- [ ] **UI** — 위 5화면에서 **동일한 간격 스케일**(예: 4의 배수 px만 사용) 위반을 **10건 이하**로 줄이기 위한 수정 PR 1건을 낸다(변경 파일 목록 + 위반 Before/After 스크린샷 또는 DOM 클래스 기준으로 검증).
- [ ] **UX** — 로그인·회원가입·API 키 생성·첫 이벤트 전송 **4 시나리오** 각각에 대해 **에러·빈 상태·로딩** 3종 상태를 수동 재현하고, 각 상태마다 **다음 행동 1개**를 안내하는 카피 또는 버튼이 있는지 O/X 표로 남긴 뒤, X인 곳을 **최대 3건** `- [ ]` 하위 과제로 쪼개어 처리한다.
- [ ] **UX** — 폼 검증 실패 시 **첫 오류 필드로 포커스 이동**하는지 E2E 또는 수동 스크립트로 확인하고, 누락 시 수정한다(재현 단계를 `RALPH_TASK` 또는 앱 README 스모크에 한 블록 추가).
- [ ] **디자인** — `apps/open-graze`에서 **제목(h1~h3)·본문·보조** 타이포 조합을 표로 정리하고, **3회 이상 중복 정의된 유사 스타일**을 1개 토큰/유틸로 합치는 리팩터 PR을 완료한다(중복 제거 전/후 grep 결과 라인 수로 측정).
- [ ] **수익화** — 공개 플랜 문구(랜딩·대시보드 billing·README 중 실제 노출분)와 `.env.example`의 **결제·플랜 관련 주석**을 나열한 뒤, **서로 모순 0건**이 되도록 정리하고 그 diff를 커밋한다.
