## 1. API·네트워크 오류

| 시나리오 | 마찰 |
|----------|------|
| S1-1 | `/` 타임라인 `GET /api/ralph/events`가 5xx·타임아웃일 때 사용자는 “무엇을 해야 하는지”를 모른다(재시도·오프라인 안내 부재). |
| S1-2 | 대시보드 `GET /api/workspaces`가 500이면 목록 `null` 처리와 달리 빈 화면·문구 없이 멈출 수 있다. |
| S1-3 | 워크스페이스 상세에서 API 키·이벤트 목록 로드 실패 시 에러 코드만 노출되거나 일관된 복구 CTA가 없다. |
| S1-4 | `/` 타임라인에서 `fetch`가 **예외**(네트워크 끊김·JSON 파싱 실패 등)로 끝나 `data === null`이 되면, 테이블 아래 **「아직 기록이 없습니다」** 빈 카피가 뜰 수 있어 **진짜 빈 타임라인과 오류가 혼동**된다. |
| S1-5 | 수집·동기화 API가 **429**(속도 제한)일 때 UI에 “잠시 후 재시도” 없이 원시 상태코드만 보이면 사용자는 **얼마나 기다려야 하는지** 알 수 없다. |
| S1-6 | `fetch`에 **클라이언트 타임아웃**이 없으면 탭이 무한 대기처럼 보이고, 취소 버튼도 없으면 마찰이 커진다. |

### 완화 과제 (API·네트워크)

- [ ] **타임라인 API 실패** — 5xx·네트워크 오류 시 인라인 메시지 + **재시도** 버튼(지수 백오프 또는 최대 N회)을 동일 영역에 둔다.
- [ ] **타임라인 `catch` 구분** — `fetch` 예외·비 JSON 응답을 `TIMELINE_EMPTY`·정상 빈 목록과 분리하고, 빈 목록 카피는 **성공 응답 + `events.length === 0`**일 때만 쓴다.
- [ ] **워크스페이스 목록 실패** — `list === null`이 아닌 **명시적 error 상태**로 분기해, “다시 시도”와 필요 시 로그아웃/지원 링크를 보여준다.
- [ ] **에러 카피 규격** — 사용자용 문장(한글)과 `코드: …`는 개발자 확장으로 접거나 “상세” 토글 뒤로 둔다(이미 타임라인에 코드 노출 패턴이 있으면 정리).
- [ ] **429·Retry-After** — 응답 헤더 또는 본문에 대기 힌트가 있으면 카운트다운·비활성 재시도 버튼으로 반영하고, 없으면 “잠시 후 다시 시도” 고정 카피를 쓴다.
- [ ] **요청 타임아웃·취소** — 대시보드·상세의 주요 `fetch`에 `AbortSignal.timeout`(또는 동등)과 사용자 노출 메시지를 맞추고, 장시간 작업은 진행률 또는 취소 가능 UI를 검토한다.

## 2. 빈 상태 (Empty state)

| 시나리오 | 마찰 |
|----------|------|
| S2-1 | 타임라인에 이벤트가 없을 때 `TIMELINE_EMPTY`와 일반 오류가 시각적으로 구분되지 않으면 사용자가 “버그인지 설정 문제인지” 판단하기 어렵다. |
| S2-2 | `/dashboard`에 워크스페이스가 0개일 때 목록 영역이 텅 비어 **다음 행동**(만들기 폼으로 스크롤·가이드)이 약하다. |
| S2-3 | 워크스페이스 **이벤트·작업** 탭에 데이터가 없을 때 “아직 없음”과 “권한 없음·필터로 가림”을 구분하지 않으면 혼란스럽다. |
| S2-4 | 워크스페이스 상세 **작업 현황**은 첫 페치 전에도 `tasks.length === 0`이면 “아직 표시할 작업이 없습니다”가 나와 **로딩 중인지 비어 있는지** 구분이 안 된다. |

### 완화 과제 (빈 상태)

- [ ] **타임라인 빈 상태** — `TIMELINE_EMPTY` 전용 카피(수집 연결·JSONL 동기화·self-test 링크 등 **한 줄 가이드**)를 유지하고, 오류 배너와 레이아웃을 분리한다.
- [ ] **워크스페이스 0개** — 히어로/일러스트 또는 짧은 체크리스트로 “첫 워크스페이스 만들기”를 목록 위에 고정 노출한다.
- [ ] **탭별 빈 상태** — 이벤트·작업·API 키 각각에 **역할별 CTA**(예: 키 발급, `platform:self-test` 문서 링크)를 둔다.
- [ ] **작업 표 빈 카피 게이트** — 작업·이벤트·키 목록은 **해당 페치가 끝난 뒤에만** “아직 없음” 문구를 보이고, 그 전에는 스켈레톤·“불러오는 중”을 쓴다.

## 3. 로딩·중복 제출

| 시나리오 | 마찰 |
|----------|------|
| S3-1 | 로그인·워크스페이스 생성 등에서 `pending`일 때 버튼이 비활성화되지 않으면 **이중 제출**이 발생한다. |
| S3-2 | `/dashboard` 초기 로딩은 한 줄 텍스트만 있어 **스켈레톤·레이아웃 시프트** 완화가 없다. |
| S3-3 | 결제 준비(`TossBillingClient`) 로딩 중 사용자가 뒤로 가거나 새로고침하면 상태가 불명확하다. |
| S3-4 | `/dashboard` 워크스페이스 **생성 폼**은 제출 중 `pending`이 없어 네트워크가 느릴 때 **중복 생성 시도**가 나기 쉽다. |
| S3-5 | `/dashboard/[slug]` API 키 **생성 폼**도 동일하게 제출 가드가 없으면 이중 `POST`가 가능하다. |
| S3-6 | 워크스페이스 상세는 키·이벤트 배열 초기값이 `[]`라, 첫 페치 전에 **“아직 없음”으로 오해**할 수 있다(로딩 배너 없음). |

### 완화 과제 (로딩)

- [ ] **폼 제출 가드** — 로그인·워크스페이스 생성·API 키 생성·작업 저장 등 모든 mutating 폼에서 `pending` 시 **버튼 disabled + 스피너/문구**를 통일한다.
- [ ] **대시보드 첫 페인트** — 워크스페이스 카드 영역에 가벼운 스켈레톤 또는 고정 높이 플레이스홀더로 CLS를 줄인다.
- [ ] **결제 플로 로딩** — 로딩 중 이탈 시 “진행 중인 결제가 있을 수 있음” 안내 또는 billing 상태 폴링/재진입 문구를 추가한다.

## 4. 폼 검증 (클라이언트·서버)

| 시나리오 | 마찰 |
|----------|------|
| S4-1 | 워크스페이스 slug는 서버에서만 엄격 검증될 때, 사용자는 제출 후에야 규칙(소문자·하이픈 등)을 이해한다. |
| S4-2 | API가 `fieldErrors`(Zod flatten)를 돌려줄 때 화면에 `JSON.stringify`로 뭉쳐 보이면 **어느 필드가 문제인지** 읽기 어렵다. |
| S4-3 | 회원가입(`/register` 등)에서 비밀번호 정책·이메일 중복 메시지가 톤이 다르면 신뢰감이 떨어진다. |
| S4-4 | 대시보드 워크스페이스 생성에서 서버가 객체형 `error`를 돌려줄 때 `JSON.stringify`로 붉은 줄이 나오면 **필드와 연결되지 않아** 수정이 번거롭다. |
| S4-5 | 가입·로그인 제출에서 `fetch`/`signIn`이 **throw**(네트워크 끊김·예상 밖 응답)하면 `catch`가 없을 때 **사용자 메시지 없이** pending만 풀리거나 오류 경계로 넘어가 **무슨 일인지** 알기 어렵다. |
| S4-6 | 비밀번호 확인 불일치는 클라이언트에서 잡히나, **이름·이메일 길이 상한**을 입력 중에 안 보여주면 서버 거절 전까지 규칙을 모른다. |

### 완화 과제 (폼 검증)

- [ ] **slug 실시간 힌트** — 입력 중 소문자 정규화 미리보기 + 규칙 1줄(`2~64자`, 허용 문자)을 필드 아래에 고정한다.
- [ ] **필드 에러 매핑** — `fieldErrors`를 필드별 `<p>`로 매핑하는 공통 헬퍼를 두고, 알 수 없는 키만 JSON fallback 한다.
- [ ] **회원가입 검증 일관성** — 클라이언트 `required`/`type=email`과 서버 메시지(중복 이메일·비밀번호 길이)를 **같은 순서·같은 톤**으로 정리한다.
- [ ] **가입·로그인 예외 처리** — `register`·`login` `onSubmit`에 `catch`로 공통 한 줄(연결 확인 등)을 세우고, 프로덕션에서는 내부 스택을 노출하지 않는다.
- [ ] **문서·랜딩 URL 정합** — 공개 문서에 적힌 가입·로그인·대시보드 URL이 `app/` 라우트와 다르면 **링크 또는 문구**를 즉시 맞춘다(요구사항을 단일 근원으로 삼는다).
- [ ] **필드 상한 힌트** — 이름·이메일·비밀번호에 서버와 동일한 **maxLength·문자 수 카운터**(선택)를 두어 제출 전 거절을 줄인다.

## 5. 성공·실패 피드백

| 시나리오 | 마찰 |
|----------|------|
| S5-1 | 워크스페이스 생성 성공 시 바로 `router.push`만 되면 “저장됨” 피드백이 없어 **느린 네트워크**에서 불안하다. |
| S5-2 | API 키를 한 번만 보여주는 패턴에서 복사 성공 여부가 없으면 사용자가 다시 생성하게 된다. |
| S5-3 | 결제 실패·웹훅 불일치 시 메시지가 기술적이면 **다음 액션**(재시도·청구 페이지)이 불분명하다. |
| S5-4 | API 키 **삭제**(`confirm` 후 `DELETE`)가 네트워크·서버 오류로 실패해도 UI에 남으면 사용자는 **삭제됐다고 착각**할 수 있다. |
| S5-5 | 로그인 폼은 자격 증명 오류 문구가 있으나, **`signIn` 네트워크 실패**와 구분되지 않으면 같은 문구로 덮여 디버깅이 어렵다. |
| S5-6 | 가입 성공 후 자동 로그인 실패 시 문구는 있으나, **로그인으로 이동하는 CTA**(이메일 프리필 `?email=` 등)가 없으면 한 단계 더 입력해야 한다. |
| S5-7 | 워크스페이스 상세에서 이벤트·작업 중 **한쪽만** 로드 실패할 때, 성공한 쪽만 갱신되면 사용자는 **“전체가 성공했다”**고 오해할 수 있다(부분 성공 vs §9-2 연계). |

### 완화 과제 (피드백)

- [ ] **성공 토스트/배너** — 생성·저장·삭제 후 짧은 비차단 성공 메시지(또는 상단 배너)를 디자인 토큰에 맞춰 통일한다.
- [ ] **클립보드 UX** — API 키·웹훅 URL 복사에 “복사됨” 상태와 `aria-live` 영역을 둔다.
- [ ] **결제 오류 카피** — 사용자용 한 줄 + “다시 시도 / 대시보드로” CTA를 `TossBilling*` 흐름에 맞춘다.
- [ ] **삭제·폐기 API 피드백** — API 키 삭제 등 `confirm` 이후 요청이 실패하면 토스트/인라인으로 **실패 사유 + 목록 새로고침**을 보여준다.
- [ ] **로그인 네트워크 구분** — `fetch`/Credentials 오류와 “비밀번호 불일치”를 사용자용으로 분리(예: “연결을 확인해 주세요”)하되 보안상 이메일 존재 여부는 노출하지 않는다.
- [ ] **가입 후 자동 로그인 실패 복구** — 실패 시 `/login?registered=0&email=…`(또는 동등)로 이동해 재입력을 최소화하고, 성공 배너와 구분한다.
- [ ] **부분 로드 성공 고지** — 키·이벤트·작업 중 일부만 실패하면 상단 또는 해당 섹션에 **“일부만 불러왔습니다”** 한 줄을 띄워 §9-2와 톤을 맞춘다.

## 6. 뒤로가기·딥링크·세션

| 시나리오 | 마찰 |
|----------|------|
| S6-1 | 로그인 `callbackUrl`이 잘못되었거나 만료된 세션으로 보호 페이지에 들어갔을 때, 루프 리다이렉트나 “왜 여기인지” 설명이 부족하다. |
| S6-2 | 브라우저 **뒤로 가기**로 결제 위젯 이전 단계로 오면 **중복 결제 시도** 위험이 있다. |
| S6-3 | 워크스페이스 slug 변경·삭제 후 북마크는 404인데 **친화적 404**(대시보드 링크)가 약할 수 있다. |
| S6-4 | 로그인 성공 직후 브라우저 **뒤로 가기**로 로그인 페이지에 돌아오면, 세션은 살아 있는데 폼이 다시 보여 **“로그인됐는지”** 혼란스럽다. |
| S6-5 | `/dashboard/[slug]/billing` 등 다단계 화면에서 **히스토리 스택**만으로 이전 단계를 복원하면 위젯·주문 상태가 어긋날 수 있다. |
| S6-6 | 워크스페이스 상세 등에서 401 시 **`router.replace("/login")`만** 호출하고 `callbackUrl`에 현재 경로를 넣지 않으면, 재로그인 후 **같은 slug 상세**로 자동 복귀하기 어렵다(코드 기준 확인 대상). |
| S6-7 | 전역 `not-found`가 **홈으로**만 안내하면, 대시보드에서 북마크한 깊은 URL을 잃은 사용자는 **한 단계 더** 거쳐야 목록에 도달한다. |
| S6-8 | 브라우저 **bfcache**로 이전 페이지에 복귀하면 클라이언트 상태는 살아 있으나 **서버 데이터는 변했을** 수 있어, stale 표시·재검증 없이 뒤로가기만 믿으면 잘못된 결정을 내릴 수 있다. |
| S6-9 | 모바일 OS **스와이프 뒤로**·제스처가 `popstate`와 겹칠 때, 결제·폼 중간 단계에서 **의도치 않은 이탈**이 잦다. |

### 완화 과제 (뒤로가기·세션)

- [ ] **callbackUrl 검증** — 상대 경로만 허용하거나 허용 목록을 두고, 거부 시 안전한 기본(`/dashboard`) + 한 줄 이유를 표시한다.
- [ ] **결제 단계 idempotency** — 동일 `orderId` 재진입 시 서버 상태를 조회해 “이미 완료/진행 중” UI를 보여준다(백엔드·프론트 공동).
- [ ] **404·403 템플릿** — 워크스페이스 범위에서 “목록으로 / 로그인” 링크를 공통 레이아웃에 둔다.
- [ ] **401 + callbackUrl** — 보호 페이지 `fetch`가 401일 때 `router.replace("/login?callbackUrl=" + encodeURIComponent(현재 경로))` 등으로 **딥링크 복귀**를 보장한다(무한 루프·오픈 리다이렉트만 검증).
- [ ] **404 보조 내비** — `not-found`에 **대시보드** 링크를 홈과 나란히 두어 관측 허브 사용자의 복귀 경로를 짧게 한다.
- [ ] **로그인 페이지 세션 인지** — 이미 세션이 있으면 “대시보드로 계속” 배너 또는 자동 리다이렉트(무한 루프만 주의)로 뒤로가기 혼란을 줄인다.
- [ ] **결제·빌링 히스토리 가드** — `popstate` 또는 라우트 진입 시 주문/세션 상태를 재검증하고, 불일치면 안전한 단계로 되돌린다.
- [ ] **bfcache·가시성 복귀** — `pageshow` `persisted` 또는 `visibilitychange`로 복귀 시 **민감 목록**(키·결제 상태)은 `revalidate`/소프트 리프레시를 검토한다.
- [ ] **모바일 뒤로 제스처** — 다단계 폼·위젯에서 단계 이탈 시 확인 시트 또는 “변경사항이 저장되지 않을 수 있음”을 정책으로 정한다.

## 9. 초기 로딩과 빈 목록 구분 (스태일·깜빡임)

| 시나리오 | 마찰 |
|----------|------|
| S9-1 | 워크스페이스 상세에서 이벤트·키 목록이 **페치 완료 전 `[]`**이면 “연동이 안 된 것”으로 오해하기 쉽다. |
| S9-2 | `Promise.all` 중 하나만 실패하면 나머지는 갱신되는데, **부분 실패**가 UI에 드러나지 않으면 데이터가 잘린 것처럼 보인다. |
| S9-3 | `/` 타임라인에서 **수동 새로고침** 없이 실시간 갱신이 없으면, 긴 세션 중 사용자는 “멈췄다”고 느낄 수 있다(제품 정책과 함께 문서화). |

### 완화 과제 (로딩 vs 빈 상태)

- [ ] **상세 페이지 로딩 플래그** — 키·이벤트 각각 또는 통합 `loading`/`error` 상태로 첫 페치 전에는 스켈레톤/스피너, 완료 후에만 빈 상태 카피를 쓴다.
- [ ] **부분 실패 표시** — 병렬 로드 중 한 API만 실패하면 해당 섹션에만 재시도 CTA를 둔다.
- [ ] **타임라인 갱신 정책** — 폴링·`revalidate`·수동 새로고침 버튼 중 하나를 선택해, 화면에 “마지막으로 불러온 시각”을 짧게 표시한다.

## 11. Ralph·JSONL·수집 연동 (운영·개발자 마찰)

| 시나리오 | 마찰 |
|----------|------|
| S11-1 | `POST /api/ralph/sync-jsonl`이 인증 실패·경로 없음·빈 JSONL 등으로 끝날 때, **HTTP 본문만**으로는 “다음에 무엇을 고치면 되는지”가 CLI·에이전트 사용자에게 분산된다. |
| S11-2 | 외부에서 `POST /api/v1/events`로 **401·400**을 받을 때 대시보드 UI에는 흔적이 없어, 키 만료·본문 스키마 문제를 **로그와 문서 사이에서** 추측해야 한다. |
| S11-3 | 홈 타임라인 카피는 “로컬 로그 반영”을 말하지만, **sync 엔드포인트 이름·필수 헤더**가 한 화면에 모이지 않으면 첫 연동까지 클릭이 많다. |

### 완화 과제 (Ralph·수집)

- [ ] **sync-jsonl 오류 카탈로그** — 대표 실패 3종(인증·파일·본문)마다 문서 또는 응답에 **한 줄 복구 문**(예: 토큰 재발급·경로·샘플 curl)을 고정한다.
- [ ] **ingest 실패 플레이북** — `401`/`400` 각각에 대해 대시보드 **API 키 영역** 또는 `/llms.txt` 링크 옆에 “자주 나는 원인” 1줄을 둔다(보안상 키 노출 없이).
- [ ] **홈 → 연동 CTA** — “로컬 로그 반영” 행에 sync API 경로 또는 문서 앵커를 **한 클릭**으로 연결한다.

## 12. 세션 만료·로그아웃·다중 탭

| 시나리오 | 마찰 |
|----------|------|
| S12-1 | JWT `maxAge`(예: 7일) 이후 첫 `fetch`가 401이면 사용자는 **입력 중이던 폼·탭 상태**를 잃고, 원인이 “만료”인지 “권한”인지 문구만으로 구분하기 어렵다. |
| S12-2 | 대시보드에서 **로그아웃**이 확인 없이 `signOut({ callbackUrl: "/" })`이면, 실수 클릭 시 작업 맥락이 한 번에 사라진다. |
| S12-3 | 한 탭에서 로그아웃했을 때 다른 탭은 **스테일 UI**로 남아, 저장·삭제 시에야 비로소 401이나 이상 동작이 드러난다(S7-1과 연결). |
| S12-4 | 결제 실패 페이지(`billing/fail`)는 `Suspense` 폴백이 **한 줄 텍스트**뿐이라, 느린 환경에서 본문과 톤이 어긋난다. |

### 완화 과제 (세션·로그아웃)

- [ ] **401 + 만료 구분** — 응답 본문 또는 전용 헤더로 “세션 만료”를 알 수 있으면 카피를 분리하고, 없으면 “다시 로그인” CTA와 **임시 초안 로컬 저장**(민감 필드 제외) 여부를 정책으로 정한다.
- [ ] **로그아웃 확인** — 최소 1회 확인(모달 또는 “5초 내 취소” 토스트) 또는 되돌리기 링크 중 하나를 적용한다.
- [ ] **크로스 탭 로그아웃** — `BroadcastChannel` 또는 `storage` 이벤트로 로그아웃을 브로드캐스트하고, 다른 탭은 즉시 로그인 유도 또는 읽기 전용 배너를 띄운다.
- [ ] **빌링 fail Suspense** — `fallback`을 대시보드와 동일 톤의 짧은 스켈레톤·스피너로 통일하고 “로딩…” 단독 문구 의존을 줄인다.

## 13. 하드 리프레시·URL 직접 입력·스크롤 복원

| 시나리오 | 마찰 |
|----------|------|
| S13-1 | 사용자가 주소창에 오타·옛 slug를 넣어 들어오면 404인데, **왜 안 되는지**(삭제됨 vs 오타) 한 줄 설명이 없으면 재시도 비용이 크다. |
| S13-2 | 긴 대시보드 폼 작성 후 **브라우저 새로고침**하면 클라이언트 상태가 초기화되어, 서버에 아직 없는 작업은 **복구 불가**다. |
| S13-3 | 성공 페이지(`billing/success`)에서 **뒤로 가기**로 결제 위젯으로 돌아가면 S6-2와 겹치며, “이미 처리됨” 배너가 없으면 중복 시도 우려가 남는다. |

### 완화 과제 (내비·복구)

- [ ] **404 slug 맥락** — 워크스페이스 404에 “목록에서 다시 선택” + slug 한 줄 표시(민감 정보 없음)로 오타 자가 점검을 돕는다.
- [ ] **폼 초안(선택)** — 워크스페이스 생성 등 긴 폼은 `sessionStorage`에 debounce 저장 또는 “나가기 전 경고” `beforeunload` 정책을 제한적으로 적용한다(모바일·PWA 이슈 검토).
- [ ] **결제 성공 후 히스토리** — 성공 직후 `replaceState`로 위젯 단계를 스택에서 대체하거나, 재진입 시 서버 조회로 “완료됨” 전용 UI를 보여 S13-3을 완화한다.

## 14. 조회 전용 UI vs “여기서 바꾸고 싶다” 기대

| 시나리오 | 마찰 |
|----------|------|
| S14-1 | 요구사항상 워크스페이스 **대시보드·상세는 조회 전용**인데, 작업(`WorkspaceTask`) 표에 상태 배지만 있으면 사용자는 **상태를 바꾸는 UI가 없다**고 느끼고 API 문서·외부 도구로 이탈한다. |
| S14-2 | “API로만 반영”이 맞더라도, **첫 성공 경로**(예: `PATCH …/tasks` 샘플·self-test와 연결)가 화면에서 한 눈에 보이지 않으면 **첫 가치 도달 클릭 수**가 늘어난다. |
| S14-3 | 작업 설명이 길 때 표 셀만으로는 잘려 **전체 맥락**을 보기 어렵고, “블로킹 사유” 같은 부가 정보가 없으면 `blocked` 상태가 정적 배지로만 남는다. |

### 완화 과제 (조회 전용·온보딩)

- [ ] **조회 전용 고지** — 작업·이벤트 섹션 상단에 “표시만: 생성·상태 변경은 API” **한 줄**과 `apps/open-graze/README.md` 또는 `/llms.txt` 앵커 링크를 둔다.
- [ ] **첫 PATCH 가이드** — 개발자용으로 `curl` 또는 HTTP 클라이언트 스니펫(민감 토큰 없이 플레이스홀더)을 **복사 가능 블록**으로 노출하거나 문서로 한 클릭 연결한다.
- [ ] **긴 제목·설명** — 행 클릭 시 모달·툴팁·접이식으로 전체 텍스트를 보여 `blocked`/`in_progress` 맥락을 읽을 수 있게 한다.

## 15. 토스트·배너·연속 피드백

| 시나리오 | 마찰 |
|----------|------|

| S15-2 | 동일 화면에 **인라인 에러**(빨간 줄)와 **상단 배너**가 동시에 있으면 어느 것을 먼저 고쳐야 할지 우선순위가 불명확하다. |

- [ ] **토스트 + 라이브 리전** — 자동 사라짐과 별도로 `aria-live` 요약 1회 또는 “활동” 패널에 최근 N건을 적재하는 옵션을 검토한다.

| 시나리오 | 마찰 |
|----------|------|
| S16-1 | 수집 이벤트 `data` JSON이 길면 **가로 스크롤**만 생기고, 모바일에서 행 높이가 커져 **다음 행과의 관계**가 끊긴다. |
| S16-2 | 타임라인·이벤트 테이블에 **고정 최소 너비**가 있으면 320px 전후에서 **전체 레이아웃이 깨지거나** 양옆이 잘린다. |
| S16-3 | 코드·JSON 블록의 **가로 스크롤**이 `main` 폭을 넘기면 사용자는 “페이지 전체가 움직인다”고 느낄 수 있다. |

### 완화 과제 (반응형·데이터 밀도)

- [ ] **JSON 접기** — 기본은 요약(키 개수·`kind` 한 줄)만 보이고, “펼치기”로 전체 `data`를 토글한다(타임라인·대시보드 이벤트 공통 패턴 검토).
- [ ] **표 반응형** — `overflow-x: auto`를 카드 단위로 감싸고, 모바일에서는 **카드 스택** 레이아웃으로 전환하는 브레이크포인트를 정한다.
- [ ] **프리 태그 래핑** — 긴 문자열은 `word-break`/`overflow-wrap` 정책을 문서화하고, 복사 버튼으로 가로 스크롤 의존을 줄인다.

## 17. 수익화·가격·문서 표면 간 정합 (UI · README · `.env.example`)

동종 SaaS는 보통 **플랜표·체험 기간·환불·영수증/청구**를 랜딩·결제·정책 페이지에 나눠 둔다. 이 앱은 **토스 v2 단건 승인 → `subscriptionStatus`** 중심이므로, 표면(코피·README·예시 env)끼리 같은 사실을 말하는지와 “없는 기능을 암시하지 않는지”를 점검한다.

| 시나리오 | 마찰 |
|----------|------|
| S17-1 | 루트 **`README.md`**는 결제 세부를 **`apps/open-graze/README.md`**로 넘기지만, 루트 **`.env.example`**에는 수집 연동만 있고 결제 키 자리·교차 링크가 없어, “한 파일만 보고” 설정하는 동료는 앱 예시를 놓칠 수 있다. |
| S17-2 | **`apps/open-graze/.env.example`**는 `TOSS_SUBSCRIPTION_AMOUNT_KRW` 기본 100·웹훅 등이 주석으로 잘 박혀 있으나, **부가세 포함 여부·과금 주기(비월간)**는 UI·README 어디에도 없어 동종 **가격 공개** 관행과 비교하면 공백이다. |
| S17-3 | **`/api/billing/toss/confirm`** 오류 페이로드가 `Unauthorized`, `invalid body`, `order mismatch` 등 **영어**인데, 성공 리다이렉트 전용 화면이 이를 그대로 보여주면 결제 실패/승인 실패 **톤이 한글 UI와 불일치**한다. |
| S17-4 | 토스 성공 후 `router.replace(\`/dashboard/${slug}?billing=success\`)`만 하고, 워크스페이스 상세에 **성공 배너·다음 액션**이 없어 사용자는 “활성화됐는지”를 목록의 작은 라벨까지 스스로 찾아야 한다. |
| S17-5 | Stripe 레거시 **`billing=cancel`** 쿼리는 URL에 남을 수 있는데, 대시보드 UI가 이를 **해석·안내하지 않으면** 취소 피드백이 비어 있다. |
| S17-6 | **무료 체험·다중 플랜·환불 절차·영수증/인보이스**는 제품에 없거나 API 수준이다. 문서/UI가 침묵하면 사용자는 동종 제품과 같은 권리가 있다고 **기대**할 수 있다. |
| S17-7 | **`/dashboard` 목록**의 `구독 활성`·`구독 비활성`은 월 정기 구독을 연상시키기 쉬운데, 결제 UI·README는 **단건 승인 후 상태 플래그**임을 분명히 말한다. 표기만 다르면 동종 대비 시 **제품 정의가 세 갈래**로 보인다. |
| S17-8 | **`POST /api/billing/toss/prepare`**는 미설정 시 한글 안내를 주지만, `Unauthorized`·`Forbidden`·`slug required` 등은 **영어**라 `TossBillingClient` 첫 화면 에러가 confirm·실패 페이지와 **톤이 갈린다**. |
| S17-9 | **루트 `README.md`**는 토스·Stripe를 한 줄로만 언급하고, **`TOSS_SUBSCRIPTION_AMOUNT_KRW` 기본 100원·단건·비월간** 같은 사실은 `apps/open-graze/README.md` 너머에만 있다. 루트만 읽는 사람은 **가격·과금 모델**을 놓친다. |
| S17-10 | **`public/llms.txt`**는 결제를 앱 README로만 넘기고, **무료 체험 없음·환불 UI 없음·단건**을 짧게 밝히지 않는다. 봇·LLM이 동종 SaaS 가정으로 답할 여지가 있다. |
| S17-11 | 토스 주문명(`prepare`의 `orderName`: `OpenGraze 워크스페이스 {slug} 구독`)은 위젯·영수증에 노출될 수 있는데, **「구독」**만으로는 README의 **구독(단건)** 설명과 자동으로 이어지지 않는다(결제 화면 `orderName`과 문서 용어 정합). |

### 완화 과제 (수익화 문구·표면 정합)

- [ ] **루트 `.env.example` ↔ 앱 예시** — 병합 정책이 허용될 때, 결제·Auth는 `apps/open-graze/.env.example`가 단일 근원임을 **한 줄 주석**으로 박거나, 루트 README와 동일한 위임 문구를 넣어 표면을 맞춘다.
- [ ] **가격·세금 표기** — `TOSS_SUBSCRIPTION_AMOUNT_KRW` 금액이 **세전/세후·부가세 별도**인지 운영 정책을 정한 뒤, 결제 화면 한 줄·`apps/open-graze/README.md` 환경 표에 동일 문구로 반영한다(동종 대비 공백 해소).
- [ ] **승인 API 사용자 메시지** — `confirm` 응답의 `error`/`detail`을 **한글 요약 + 개발자용 원문(접기)**로 매핑하거나, 서버에서부터 사용자용 코드를 내려 `TossBillingSuccess` 톤을 `TossBillingClient`·실패 페이지와 맞춘다.
- [ ] **`?billing=success` 배너** — 워크스페이스 상세(또는 상단 레이아웃)에서 쿼리를 읽어 “구독이 활성화되었습니다” 등 **비모달 한 줄**을 보여 준 뒤 `replaceState`로 쿼리를 지운다(§5·§13과 중복 시 하나로 합친다).
- [ ] **`?billing=cancel` (Stripe)** — 레거시 플로를 유지한다면 동일 패턴으로 취소 안내를 띄우고, 제거한다면 README·환경 문서에서 cancel URL 기대를 삭제한다.
- [ ] **단일 상품·비체험 명시** — 플랜 티어가 없으면 랜딩·README·결제 상단 중 한 곳에 **「워크스페이스당 단건 결제·무료 체험 구간 없음(또는 inactive 시 …)」**을 한 문장으로 고정한다.
- [ ] **환불·영수증 정책** — 없으면 “미제공” 또는 외부 정책 URL을 한 줄로라도 README·앱 내 링크에 두어 청구·환불 질의를 동종 수준으로 흡수한다.
- [ ] **워크스페이스 목록 `subscriptionStatus` 카피** — `구독 활성` 등을 **단건 결제로 활성화됨**·보조 설명 툴팁 등으로 바꾸거나, 결제 페이지·README와 동일한 용어 표를 쓴다(S17-7).
- [ ] **prepare API 에러 한글화** — `Unauthorized`·`Forbidden`·`slug required`·`Invalid JSON` 등을 사용자용 한글로 통일하고, 개발자용 원문은 접거나 로그로만 남긴다(S17-8).
- [ ] **루트 README 수익화 한 줄** — 앱 README로 위임하되, **단건·비월간·금액 env 기본값**을 중복 없이 한 문장으로 요약해 동종 **가격 공개** 기대를 맞춘다(S17-9).
- [ ] **`/llms.txt` 수익화 각주** — 짧은 인덱스에 「결제는 워크스페이스 단위 단건·무료 체험·인앱 환불 절차 없음 → 상세는 앱 README」를 넣을지 결정하고 문구를 고정한다(S17-10).
- [ ] **토스 `orderName` vs 문서** — 위젯·매출전표에 보이는 문자열이 **구독(단건)** 과 모순되지 않게 `orderName` 또는 README 중 한쪽을 조정한다(S17-11).

## 18. 글로벌 예외·오류 경계·클라이언트 크래시

| 시나리오 | 마찰 |
|----------|------|
| S18-1 | 레이아웃·페이지에서 **처리되지 않은 런타임 예외**가 나면 Next 기본 오류 화면만 보이고, **대시보드·로그아웃**으로 돌아가는 일관 경로가 없을 수 있다. |
| S18-2 | 클라이언트 컴포넌트 `hydration` 불일치 경고만 콘솔에 있고 UI는 깨져 보이면 사용자는 **“내 데이터가 망가졌는지”** 오해한다. |
| S18-3 | 서드파티 스크립트·결제 위젯이 실패해도 **나머지 페이지가 쓸모 있게** 남아야 하는데, 한 컴포넌트 예외가 전체를 죽이면 마찰이 크다. |

### 완화 과제 (오류 경계)

- [ ] **루트·세그먼트 `error.tsx`** — 복구용 “다시 시도”·홈·대시보드 링크를 공통 톤으로 두고, 프로덕션에서는 스택 대신 **요청 ID**(있을 때)만 노출한다.
- [ ] **위젯·결제 격리** — 토스 등 외부 위젯은 `ErrorBoundary`로 감싸 실패 시 “결제만 일시 불가” 카드로 낮춘다.
- [ ] **hydration 이슈 대응** — 개발에서 원인 수정·프로덕션에서 `suppressHydrationWarning` 남용을 피하고, 날짜·랜덤 ID 등은 클라이언트 전용 마운트 패턴으로 통일한다.

## 19. 온보딩·환경 디폴트·문서 링크·대량 타임라인 스캔

로컬 시드·개발 편의와 **실제 배포 사용자** 기대가 겹칠 때, 오류/빈 상태가 아니어도 **첫 방문·첫 로그인**에서 마찰이 생긴다. 동종 관측 SaaS는 보통 랜딩·앱 내 도움말이 **배포 URL 기준**으로 동작한다.

| 시나리오 | 마찰 |
|----------|------|
| S19-1 | `/login` 이메일 입력 기본값이 **시드 계정 주소**로 고정되면, 비로컬·신규 사용자는 “이미 누군가 계정이 채워져 있다”고 오해하거나 매번 지워야 한다. |
| S19-2 | 로그인·가입 카피가 **저장소 상대 경로**(`docs/…`, `apps/…`)만 가리키면, 호스팅된 앱만 쓰는 사용자는 **클릭 한 번에 도움말**에 닿기 어렵다. |
| S19-3 | 가입 직후 안내 배너(“가입이 완료되었습니다…”)가 **URL 쿼리·상태와 연결되지 않으면** 표시되지 않거나, 구현과 카피가 어긋난 채로 남는다. |
| S19-4 | 비밀번호 필드에 **Caps Lock**·**표시 토글** 힌트가 없으면 오타·대소문자 실수로 같은 오류 메시지만 반복된다. |
| S19-5 | `signIn`/`fetch`가 **네트워크 예외**로 실패할 때 `catch` 없이 `finally`만 있으면 사용자는 “버튼이 잠깐 눌렸다” 수준으로만 느낀다(§4-5·§5-5와 합쳐서 한 번에 고칠 것). |
| S19-6 | `/` 타임라인에 **종류·역할·소스 필터**나 검색이 없으면 이벤트가 수백 건일 때 **스크롤만으로 역할별 스캔**이 어렵다(북극성 “역할별 모니터링”과 직결). |
| S19-7 | 대시보드 **워크스페이스 카드**에 구독·결제 상태가 있을 때, 결제 미설치·비활성 워크스페이스가 **동일한 시각 무게**로 나오면 “지금 결제해야 하는지” 우선순위가 불명확하다. |
| S19-8 | 긴 세션에서 **탭 제목·파비콘 배지**(미읽음·실패)가 없으면 백그라운드 탭만으로는 “수집이 멈췄는지” 알기 어렵다. |

### 완화 과제 (온보딩·환경)

- [ ] **로그인 이메일 기본값** — 프로덕션 빌드에서는 빈 문자열(또는 `callbackUrl`로 유입 시에만 프리필)로 두고, 로컬·시드 안내는 힌트 문구·`placeholder`로만 노출한다(S19-1).
- [ ] **배포 기준 문서 링크** — 공개 문서가 같은 오리진에 없다면, `/llms.txt`·`/api/v1/meta/limits` 등 **앱 안 URL**을 우선 링크하고 저장소 경로는 “소스에서 읽을 때” 보조로 둔다(S19-2).
- [ ] **가입 완료 배너 연동** — `?registered=1` 등 쿼리와 `RegisterInner`의 자동 로그인 실패 시 `router.push("/login?…")`를 맞추고, 배너 표시 조건을 한 곳에서 읽는다(S19-3).
- [ ] **비밀번호 입력 보조** — `type="password"` 옆에 표시 토글(접근 가능한 버튼)·Caps Lock 감지 시 한 줄 힌트(가능한 브라우저만)를 검토한다(S19-4).
- [ ] **로그인·가입 submit `catch`** — `signIn`·`fetch` 모두에서 네트워크 실패용 한 줄을 §4·§5와 동일 톤으로 세운다(S19-5).
- [ ] **타임라인 필터 MVP** — `kind`·`detail.role`·`source` 중 최소 1축 필터 또는 클라이언트 검색으로 “역할별 스캔” 비용을 줄인다(S19-6).
- [ ] **워크스페이스 카드 상태 계층** — 결제 미설정·inactive는 보조 색·접기 또는 툴팁으로 “지금 할 일” 우선순위를 시각적으로 내린다(S19-7).
- [ ] **탭 가시성(선택)** — 수집 실패·결제 실패 시 `document.title` 접두사 `(1)` 또는 짧은 상태 접두를 정책으로 둘지 검토한다(S19-8).

## 유지 방법

- 동종 SaaS 비교 후 새 마찰이 나오면 **시나리오 표 한 줄 + `- [ ]` 1개 이상**을 같은 형식으로 이 문서에 추가한다. 상단 **마찰 축 → 절 색인** 표에 행을 더해 탐색을 유지한다.
- 구현이 끝난 `- [ ]`는 구현 담당 이터에서 `[x]`로 바꾸고, 필요하면 `RALPH_TASK.md` 하위 항목과 동기화한다.
